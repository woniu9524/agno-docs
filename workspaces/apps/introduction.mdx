Apps は、`FastAPI`、`PgVector`、`Streamlit`、`Jupyter`、`Django` のようなツールであり、これらを Python クラスとして定義し、`ag start` または `ag ws up` を使用して実行します。

agno で Apps を実行する場合、それらをより高レベルの抽象化におけるインフラストラクチャ・アズ・コードと考えてください。コンテナやボリュームなどを定義する代わりに、実行したいアプリケーションを定義します。インフラストラクチャ・アズ・コードではなく、**アプリケーション・アズ・コード**を実行します。

同じ `App` を docker、AWS (ECS)、または Kubernetes (EKS) で実行できます。App は、LoadBalancer、Service、Deployment のような基盤となるリソースを作成します。基盤となるリソースがより複雑になるにつれて、Apps の概念はより魅力的になります。

## Example

Jupyter notebook と PgVector を docker 上で実行してみましょう。

以下の内容を `resources.py` というファイルにコピーし、`ag start resources.py` を実行してください。

```python resources.py
from agno.docker.app.jupyter import Jupyter
from agno.docker.app.postgres import PgVectorDb
from agno.docker.resources import DockerResources

# -*- PgVector running on port 5432:5432
vector_db = PgVectorDb(pg_user="ai", pg_password="ai", pg_database="ai")

# -*- Jupyter running on port 8888:8888
jupyter = Jupyter(mount_workspace=True)

# -*- DockerResources
dev_docker_resources = DockerResources(apps=[vector_db, jupyter])
```

- 各 App は、入力と型の検証を提供する pydantic オブジェクトです。
- `mount_workspace` がディレクトリを自動的にマウントする方法に注意してください。
- `PgVectorDb` が必要な設定を行い、ボリュームを作成する方法に注意してください。

これは簡単な例ですが、これらの概念は複雑なアプリケーションにとって非常に強力になります。

## Motivation

Apps は、AI 製品の**「アプリケーションレイヤー」**を提供します。

私たちが書くソフトウェアはアプリケーションによって提供される必要があり、このアプリケーションは開発時にはローカルで、本番環境ではクラウドで**同じように実行**される必要があります。**アプリケーション・アズ・コード**を定義することで、インフラストラクチャ・アズ・コードの利点をソフトウェアレイヤーにもたらします。

また、**アプリケーション・アズ・コード**を定義することで、「ソフトウェアシステム」をテンプレートにパッケージ化できます。これは、すべての agno テンプレートが docker を使用してローカルで、または 1 コマンドで AWS 上で実行できることを意味します。

最後に、**アプリケーションを Python オブジェクトとして**定義することは、通常のオブジェクトのようにコードにインポートできることを意味し、以下のコードを可能にします。

```python
from resources import vector_db

db_url=vector_db.get_db_connection_local()
```
