---
title: セッションストレージ
---

**セッションストレージ**を使用して、エージェントのセッションと状態をデータベースまたはファイルに永続化します。

<Tip>
**なぜセッションストレージが必要なのか？**

エージェントは一時的なものであり、組み込みメモリは現在の実行サイクルでのみ持続します。

本番環境では、API経由でエージェントを提供（またはトリガー）し、複数のリクエスト間で同じセッションを継続する必要があります。ストレージは、セッション履歴と状態をデータベースに永続化し、中断したところから再開できるようにします。

ストレージを使用すると、エージェントセッションを検査および評価し、few-shotの例を抽出し、内部監視ツールを構築することもできます。これにより、**データを参照する**ことができ、より良いエージェントを構築するのに役立ちます。
</Tip>

Agent、Team、またはWorkflowにストレージを追加することは、`Storage`ドライバーを提供するのと同じくらい簡単です。Agnoが残りを処理します。Sqlite、Postgres、Mongo、またはその他の任意のデータベースを使用できます。

実行サイクル間で永続化を示す簡単な例を次に示します。

```python storage.py
from agno.agent import Agent
from agno.models.openai import OpenAIChat
from agno.storage.sqlite import SqliteStorage
from rich.pretty import pprint

agent = Agent(
    model=OpenAIChat(id="gpt-4o-mini"),
    # 実行サイクル間で同じセッションを継続するためにセッションIDを固定します
    session_id="fixed_id_for_demo",
    storage=SqliteStorage(table_name="agent_sessions", db_file="tmp/data.db"),
    add_history_to_messages=True,
    num_history_runs=3,
)
agent.print_response("私の最後の質問は何でしたか？")
agent.print_response("フランスの首都はどこですか？")
agent.print_response("私の最後の質問は何でしたか？")
pprint(agent.get_messages_for_session())
```

これを最初に実行したとき、「私の最後の質問は何でしたか？」という質問への回答は利用できません。しかし、もう一度実行すると、エージェントは適切に回答できるようになります。セッションIDを固定したため、スクリプトを実行するたびにエージェントは同じセッションから再開します。

## ストレージのメリット

ストレージは通常、エージェントエンジニアリングにおいてあまり議論されない部分ですが、本番環境のエージェントアプリケーションの「縁の下の力持ち」であると私たちは考えています。

本番環境では、ストレージは以下に必要です。

- セッションの継続：セッション履歴を取得し、中断したところから再開します。
- セッションリストの取得：以前のセッションを継続するには、そのエージェントで利用可能なセッションのリストを維持する必要があります。
- 実行間の状態の保存：エージェントの状態をデータベースまたはファイルに保存して、後で検査できるようにします。

しかし、それだけではありません。

- ストレージは、エージェントのセッションデータを検査および評価のために保存します。
- ストレージは、エージェントの改善に使用できるfew-shotの例を抽出するのに役立ちます。
- ストレージにより、内部監視ツールやダッシュボードを構築できます。

<Warning>
ストレージはエージェントインフラストラクチャの非常に重要な部分であるため、サードパーティにオフロードすることは決してありません。エージェントには、ほぼ常に独自のストレージレイヤーを使用する必要があります。
</Warning>

## 例：Postgresをストレージとして使用する

<Steps>
  <Step title="Postgresの実行">

    [Docker Desktop](https://docs.docker.com/desktop/install/mac-install/)をインストールし、次のコマンドを使用してポート**5532**で**Postgres**を実行します。

    ```bash
    docker run -d \
      -e POSTGRES_DB=ai \
      -e POSTGRES_USER=ai \
      -e POSTGRES_PASSWORD=ai \
      -e PGDATA=/var/lib/postgresql/data/pgdata \
      -v pgvolume:/var/lib/postgresql/data \
      -p 5532:5432 \
      --name pgvector \
      agno/pgvector:16
    ```

  </Step>
  <Step title="ストレージを持つエージェントの作成">
    `agent_with_storage.py`という名前のファイルを作成し、以下の内容を記述します。

    ```python
    import typer
    from typing import Optional, List
    from agno.agent import Agent
    from agno.storage.postgres import PostgresStorage
    from agno.knowledge.pdf_url import PDFUrlKnowledgeBase
    from agno.vectordb.pgvector import PgVector, SearchType

    db_url = "postgresql+psycopg://ai:ai@localhost:5532/ai"
    knowledge_base = PDFUrlKnowledgeBase(
        urls=["https://agno-public.s3.amazonaws.com/recipes/ThaiRecipes.pdf"],
        vector_db=PgVector(table_name="recipes", db_url=db_url, search_type=SearchType.hybrid),
    )
    storage = PostgresStorage(table_name="pdf_agent", db_url=db_url)

    def pdf_agent(new: bool = False, user: str = "user"):
        session_id: Optional[str] = None

        if not new:
            existing_sessions: List[str] = storage.get_all_session_ids(user)
            if len(existing_sessions) > 0:
                session_id = existing_sessions[0]

        agent = Agent(
            session_id=session_id,
            user_id=user,
            knowledge=knowledge_base,
            storage=storage,
            # レスポンスでツール呼び出しを表示する
            show_tool_calls=True,
            # エージェントがチャット履歴を読み取れるようにする
            read_chat_history=True,
            # チャット履歴をモデルに送信されるメッセージに自動的に追加することもできます
            # ただし、モデルにチャット履歴を与えることが常に役立つとは限らないため、ツールとして提供します
            # 必要に応じてのみ使用します
            # add_history_to_messages=True,
            # メッセージに追加する履歴応答の数。
            # num_history_responses=3,
        )
        if session_id is None:
            session_id = agent.session_id
            print(f"開始セッション: {session_id}\n")
        else:
            print(f"継続セッション: {session_id}\n")

        # エージェントをCLIアプリとして実行します
        agent.cli_app(markdown=True)


    if __name__ == "__main__":
        # ナレッジベースをロードします: 初回実行後にコメントアウトしてください
        knowledge_base.load(upsert=True)

        typer.run(pdf_agent)
    ```

  </Step>
  <Step title="エージェントの実行">
    ライブラリをインストールします

    <CodeGroup>

    ```bash Mac
    pip install -U agno openai pgvector pypdf "psycopg[binary]" sqlalchemy
    ```

    ```bash Windows
    pip install -U agno openai pgvector pypdf "psycopg[binary]" sqlalchemy
    ```

    </CodeGroup>

    エージェントを実行します

    ```bash
    python agent_with_storage.py
    ```

    これでエージェントはセッション間で継続されます。質問をしてください。

    ```
    パッタイの作り方を教えてください？
    ```

    その後、`bye`とメッセージして終了し、アプリを再度起動して次のように質問してください。

    ```
    私の最後のメッセージは何でしたか？
    ```

  </Step>
  <Step title="新しい実行の開始">
    新しい実行を開始するには、`agent_with_storage.py`ファイルを`--new`フラグと共に実行します。

    ```bash
    python agent_with_storage.py --new
    ```

  </Step>
</Steps>

## スキーマのアップグレード

`AgentStorage`を使用する場合、SQLベースのストレージクラスには固定スキーマがあります。新しいAgno機能がリリースされると、スキーマを更新する必要がある場合があります。

アップグレードは手動または自動で行うことができます。

### 自動アップグレード

自動アップグレードは、ストレージクラスコンストラクタで`auto_upgrade_schema`パラメータが`True`に設定されている場合に行われます。
エージェント実行に対してこの設定を一度行うだけで、スキーマがアップグレードされます。

```python
db_url = "postgresql+psycopg://ai:ai@localhost:5532/ai"
storage = PostgresStorage(table_name="agent_sessions", db_url=db_url, auto_upgrade_schema=True)
```

### 手動アップグレード

手動スキーマアップグレードは、ストレージクラスの`upgrade_schema`メソッドを呼び出すことで実行できます。

```python
db_url = "postgresql+psycopg://ai:ai@localhost:5532/ai"
storage = PostgresStorage(table_name="agent_sessions", db_url=db_url)
storage.upgrade_schema()
```
